# Fix: Terminal font scaling on display changes and tab switches

**Date:** 2026-04-05
**Status:** Analysis complete, fix not yet implemented
**Branch:** `develop`

## Problem

When the app window moves between displays with different pixel densities (e.g., closing laptop lid while connected to an external monitor, or detaching the external monitor), the terminal font appears at the wrong size — too big on the new display or too small when returning. There is no way to correct it without restarting the app.

Additionally, the issue appears when switching between tabs on the same monitor — no display change needed.

## Root cause analysis (comparing with Ghostty upstream)

Reference file: `/Users/maximgorbatyuk/projects/ghostty/macos/Sources/Ghostty/Surface View/SurfaceView_AppKit.swift`

### Root Cause 1: `setSurfaceSize()` has dedup cache — Ghostty has none

**SlothyTerminal** (`GhosttySurfaceView.swift:344-357`):
```swift
private func setSurfaceSize(width: UInt32, height: UInt32) {
    guard let surface, width > 0, height > 0 else { return }
    guard surfaceMetricsCache.shouldApplySurfaceSize(width: width, height: height) else {
        return  // SKIPS the call if size unchanged
    }
    ghostty_surface_set_size(surface, width, height)
}
```

**Ghostty upstream** (`SurfaceView_AppKit.swift:489-503`):
```swift
private func setSurfaceSize(width: UInt32, height: UInt32) {
    guard let surface = self.surface else { return }
    ghostty_surface_set_size(surface, width, height)  // ALWAYS sends
    let size = ghostty_surface_size(surface)
    DispatchQueue.main.async {
        self.surfaceSize = size
    }
}
```

**Impact:** When the display changes but the framebuffer pixel dimensions happen to be the same, the size update is silently dropped. GhosttyKit may use `set_size` as a trigger to reconfigure its Metal renderer for the new display/scale combination. After tab switch, `invalidateSurfaceMetrics()` resets the cache, but the fundamental issue is that Ghostty never deduplicates — it always sends.

### Root Cause 2: `contentSize` has no fallback to `frame.size`

**SlothyTerminal** (`GhosttySurfaceView.swift:32`):
```swift
private var contentSize: NSSize = .zero
```

**Ghostty upstream** (`SurfaceView_AppKit.swift:151-155`):
```swift
private var contentSizeBacking: NSSize?
private var contentSize: NSSize {
    get { return contentSizeBacking ?? frame.size }  // falls back to frame.size
    set { contentSizeBacking = newValue }
}
```

**Impact:** If `viewDidChangeBackingProperties()` fires before the first `layout()` (e.g., during surface creation or rapid display change), SlothyTerminal uses `.zero` for the size calculation, sending a zero-size update that gets blocked by guards. Ghostty gracefully falls back to `frame.size`, ensuring a valid size is always sent. This also matters during the `handleScreenChange()` async dispatch path.

### Root Cause 3: Tab activation re-sends size but NOT content scale

**SlothyTerminal** tab switch path (`TerminalView.swift:102-104`):
```
invalidateSurfaceMetrics()  -> resets BOTH size AND content scale cache
needsLayout = true          -> layout() -> sizeDidChange() -> setSurfaceSize()
                               ^^ ONLY sends size, never content scale
```

**Ghostty upstream** has no equivalent tab-switching invalidation logic — it has no dedup cache to invalidate, and `SurfaceScrollView.layout()` always sends size unconditionally via `synchronizeCoreSurface()`.

**Impact:** The asymmetry of resetting both cache entries but only re-populating one (size) leaves the content scale cache in a cleared state. The tab activation path should match what Ghostty does naturally — send both size and scale unconditionally.

### Additional finding: Ghostty reads back `ghostty_surface_size()` after set

Ghostty's `setSurfaceSize` calls `ghostty_surface_size(surface)` after `ghostty_surface_set_size` to read back the actual grid dimensions and publishes them. SlothyTerminal doesn't do this readback. This feedback loop might be how Ghostty detects and corrects internal state mismatches.

## Trigger-behavior matrix

| Trigger | content_scale sent? | size sent? | Notes |
|---|---|---|---|
| Tab switch | NO | YES (cache was reset) | Only size re-sent, not content scale |
| Display change (different DPI) | YES (unconditional) | YES (if dimensions changed) | Size may be deduped if pixel dims match |
| Display change (same DPI) | YES (unconditional) | NO (deduped, same dims) | Dedup may prevent necessary reconfiguration |
| Window resize | NO | YES (if changed) | Normal resize path |
| Backing scale change | YES (unconditional) | YES | Full reconfiguration |

## Fix plan

### 1. Remove `GhosttySurfaceMetricsCache` entirely

**File:** `GhosttySurfaceView.swift`

- Remove `surfaceMetricsCache` property
- In `setSurfaceSize()`, remove the `shouldApplySurfaceSize` guard — always call `ghostty_surface_set_size`
- Remove `invalidateSurfaceMetrics()` method
- In `destroySurface()`, remove `surfaceMetricsCache.reset()` call

**File:** `TerminalView.swift`

- In `updateNSView()`, remove `nsView.invalidateSurfaceMetrics()` call (method no longer exists)

**File:** `GhosttySurfaceMetricsCache.swift`

- Can be deleted entirely, or kept for tests. The struct is no longer used from production code.

**Rationale:** Ghostty upstream has NO dedup cache. The cache was an optimization to avoid redundant C API calls, but it introduces subtle bugs by preventing necessary reconfiguration signals.

### 2. Change `contentSize` to use Ghostty's computed property pattern

**File:** `GhosttySurfaceView.swift`

Replace:
```swift
private var contentSize: NSSize = .zero
```

With:
```swift
private var contentSizeBacking: NSSize?
private var contentSize: NSSize {
    get { return contentSizeBacking ?? frame.size }
    set { contentSizeBacking = newValue }
}
```

**Rationale:** Ensures `viewDidChangeBackingProperties()` always has a valid size to work with, even if called before the first `layout()` pass.

### 3. In tab activation, also re-send content scale

**File:** `TerminalView.swift`

In `updateNSView()`, when tab becomes active, call `viewDidChangeBackingProperties()` (which sends both content scale and size) instead of just `needsLayout = true`:

```swift
if isActive {
    nsView.needsLayout = true
    // Also trigger a full backing-property refresh so content scale
    // is re-sent alongside size — matches Ghostty's unconditional approach.
    DispatchQueue.main.async {
        nsView.viewDidChangeBackingProperties()
    }
}
```

Or, since `viewDidChangeBackingProperties()` already calls `setSurfaceSize()`, and we no longer have dedup, just calling `viewDidChangeBackingProperties()` from the tab activation path would send both content scale and size.

**Rationale:** Ensures tab activation sends the full set of display metrics to GhosttyKit, not just size.
